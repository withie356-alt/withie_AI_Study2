[디자인씽킹 알 키우기 - 문제 은행]
아래 문제에서 영역별로 랜덤 1개씩 + 랜덤 영역 1개 = 총 6문제를 출제하세요.
4개 보기는 반드시 같은 상황 안에서 나와야 합니다.

═══════════════════════════════════
STAGE 1: 문제발견(공감)
═══════════════════════════════════

Q1-1. 병원 접수 대기 시간이 길다는 민원이 많다. 가장 먼저 해야 할 일은?
[A] 접수 키오스크를 도입한다
[B] 직원을 추가 배치한다
[C] 환자들이 대기하는 모습을 현장에서 관찰한다
[D] 대기 번호 앱을 개발한다
정답: C
해설: 문제발견은 현장에서 사용자를 직접 관찰하는 것부터 시작해요. A, B, D는 모두 원인도 모른 채 내놓는 해결책이에요!

Q1-2. 사내 식당 이용률이 떨어지고 있다. 문제를 발견하려면?
[A] 메뉴를 고급화한다
[B] 식당 인테리어를 리뉴얼한다
[C] 직원들의 점심시간 행동을 관찰하고 인터뷰한다
[D] 할인 쿠폰을 제공한다
정답: C
해설: 왜 안 오는지 이유를 먼저 파악해야 해요. 관찰과 인터뷰가 문제발견의 핵심!

Q1-3. 신입사원 이직률이 높다. 원인을 찾으려면?
[A] 연봉을 인상한다
[B] 퇴사자와 재직자를 인터뷰하고 업무 환경을 관찰한다
[C] 멘토링 프로그램을 만든다
[D] 워크숍을 개최한다
정답: B
해설: 해결책을 내놓기 전에, 실제로 무엇이 힘든지 당사자에게 물어보는 것이 먼저예요.

Q1-4. 고객센터 전화 문의가 폭주한다. 문제발견을 위한 첫 단계는?
[A] 자동응답 시스템(ARS)을 강화한다
[B] 상담사를 더 채용한다
[C] 고객이 어떤 문의를 하는지 현장에서 듣고 기록한다
[D] FAQ 페이지를 만든다
정답: C
해설: 어떤 문의가 왜 많은지 현장에서 파악해야 진짜 문제를 찾을 수 있어요.

Q1-5. 회사 내부 커뮤니케이션이 잘 안 된다는 불만이 있다. 어떻게 시작해야 할까?
[A] 새로운 메신저 도구를 도입한다
[B] 주간 회의를 늘린다
[C] 각 팀원들이 실제로 어떻게 소통하는지 하루 동안 따라가며 관찰한다
[D] 커뮤니케이션 교육을 실시한다
정답: C
해설: 도구나 제도를 바꾸기 전에, 실제 소통 방식을 관찰하는 게 문제발견의 출발점이에요.

STAGE 2: 문제정의
═══════════════════════════════════

Q2-1. 카페에서 "주문이 느리다"는 고객 불만이 있다. 문제를 제대로 정의하려면?
[A] 키오스크를 설치한다
[B] 바리스타를 추가 채용한다
[C] 고객이 "느리다"고 느끼는 순간이 정확히 언제인지 파악한다
[D] 메뉴 수를 줄인다
정답: C
해설: "느리다"의 진짜 의미를 정의해야 해요. 주문? 결제? 음료 나오는 시간? A, B, D 전부 해결책이에요!

Q2-2. 온라인 쇼핑몰 장바구니 이탈률이 높다. 문제 정의로 적절한 것은?
[A] 할인 쿠폰을 팝업으로 띄운다
[B] 결제 과정을 간소화한다
[C] "고객이 결제 직전에 망설이는 진짜 이유가 무엇인가?"로 문제를 재정의한다
[D] 무료 배송 조건을 낮춘다
정답: C
해설: 이탈하는 '진짜 이유'를 정의해야 올바른 해결책이 나와요. 나머지는 원인도 모르고 내놓는 해결책!

Q2-3. 사내 교육 만족도가 낮다. 문제 정의로 올바른 접근은?
[A] 교육 콘텐츠를 전면 리뉴얼한다
[B] 외부 전문 강사를 섭외한다
[C] "직원들이 교육에서 실제로 무엇을 기대하는데 충족이 안 되는가?"를 파악한다
[D] 교육 시간을 줄인다
정답: C
해설: 만족도가 '왜' 낮은지 정의하지 않으면, 아무리 바꿔도 같은 불만이 반복돼요.

Q2-4. 배달앱 리뷰에 "음식이 늦게 온다"는 불만이 많다. 문제를 정의하려면?
[A] 배달 기사를 늘린다
[B] 예상 도착 시간을 넉넉하게 표시한다
[C] "늦다"고 느끼는 기준이 시간인지, 기대와의 차이인지, 정보 부족인지 구분한다
[D] 음식 조리 시간을 단축한다
정답: C
해설: "늦다"는 사람마다 기준이 달라요. 진짜 문제가 시간인지 기대관리인지 먼저 정의해야 해요.

Q2-5. 회의가 비효율적이라는 불만이 많다. 문제정의를 제대로 하려면?
[A] 회의 시간을 30분으로 제한한다
[B] 화상회의 도구를 바꾼다
[C] "비효율적"이 구체적으로 무엇을 의미하는지 파악한다 - 길어서? 결론이 안 나서? 불필요해서?
[D] 회의록 양식을 만든다
정답: C
해설: "비효율적"은 모호한 표현이에요. 구체적으로 뭐가 문제인지 정의하는 게 먼저!

Q2-6. 관찰 결과 고객들이 무인 편의점에서 결제 오류로 짜증을 낸다. 올바른 문제정의는?
[A] "결제 시스템을 더 빠른 것으로 교체해야 한다"
[B] "직원을 상시 배치해야 한다"
[C] "고객이 결제 과정에서 어떤 단계에서 혼란을 느끼는가?"
[D] "결제 수단을 카드만 가능하게 단순화해야 한다"
정답: C
해설: A, B, D는 모두 해결책이에요. 문제정의는 "어디서 혼란이 생기는가?"를 밝히는 거예요.

STAGE 3: 아이데이션
═══════════════════════════════════

Q3-1. 아이데이션 단계에서 지켜야 할 가장 중요한 원칙은?
[A] 실현 가능한 아이디어만 낸다
[B] 예산 범위 안에서 생각한다
[C] 판단을 멈추고 최대한 많은 아이디어를 발산한다
[D] 팀장이 방향을 먼저 제시한다
정답: C
해설: 아이데이션은 발산의 시간! "그건 안 돼"를 멈추고 일단 많이 내는 게 핵심이에요.

Q3-2. 브레인스토밍 중 동료가 황당한 아이디어를 냈다. 어떻게 해야 할까?
[A] 현실적이지 않다고 바로 피드백한다
[B] 일단 넘어가고 무시한다
[C] "그 아이디어에 더해서..." 하고 발전시킨다
[D] 투표로 탈락시킨다
정답: C
해설: "Yes, and..." 원칙! 남의 아이디어 위에 쌓아가는 게 좋은 브레인스토밍이에요.

Q3-3. "어떻게 하면 신입사원이 현장에서 필요한 정보를 바로 찾을 수 있을까?" 이 문장의 형태는?
[A] 사용자 페르소나
[B] How Might We (HMW) 질문
[C] 프로토타입 기획서
[D] 테스트 시나리오
정답: B
해설: "어떻게 하면 ~할 수 있을까?"는 HMW(How Might We) 형태예요. 문제정의를 아이디어 발산으로 연결하는 다리 역할!

Q3-4. 아이데이션에서 나온 20개 아이디어 중 다음 단계로 갈 것을 고르려면?
[A] 가장 쉬운 것을 고른다
[B] 팀장이 결정한다
[C] 실현 가능성과 임팩트를 기준으로 팀이 함께 평가한다
[D] 가장 처음 나온 아이디어를 고른다
정답: C
해설: 수렴할 때는 기준을 정하고 팀이 함께 평가해요. 직급이나 순서가 아니라 기준!

Q3-5. 아이데이션 세션을 할 때 피해야 할 행동은?
[A] 포스트잇에 아이디어를 적어 붙인다
[B] "그건 예산이 없어서 안 돼"라고 말한다
[C] 다른 사람의 아이디어에 아이디어를 더한다
[D] 엉뚱한 아이디어도 환영한다
정답: B
해설: 발산 단계에서 "안 돼"는 금지어! 실현 가능성은 나중에 판단해요.

STAGE 4: 프로토타입
═══════════════════════════════════

Q4-1. 프로토타입의 핵심 목적은?
[A] 완성도 높은 제품을 만드는 것
[B] 투자자에게 보여주기 위한 시연용
[C] 아이디어를 빠르게 눈에 보이는 형태로 만들어 검증하는 것
[D] 개발팀에 넘길 상세 설계서를 만드는 것
정답: C
해설: 프로토타입은 "빠르게, 싸게, 대충이라도" 만들어서 검증하는 게 핵심!

Q4-2. 새로운 사내 앱 아이디어를 프로토타입으로 만들려면?
[A] 개발자에게 의뢰해서 실제 앱을 만든다
[B] 종이에 화면을 그려서 사용자에게 보여준다
[C] 기획서를 먼저 완벽하게 작성한다
[D] 시장 조사 보고서를 만든다
정답: B
해설: 종이 프로토타입(Paper Prototype)만으로도 충분히 아이디어를 검증할 수 있어요. 빠르게!

Q4-3. 프로토타입을 만들 때 가장 중요한 것은?
[A] 디자인이 예뻐야 한다
[B] 모든 기능이 작동해야 한다
[C] 검증하고 싶은 핵심 가설에 집중한다
[D] 경쟁사 제품과 비교할 수 있어야 한다
정답: C
해설: 프로토타입은 "이게 될까?"를 확인하는 거예요. 완성도가 아니라 가설 검증이 목적!

Q4-4. 프로토타입에 대한 설명으로 틀린 것은?
[A] 실패해도 괜찮다, 빨리 배우는 게 목적이다
[B] 완벽할수록 좋다
[C] 하루 만에 만들 수 있는 수준이면 충분하다
[D] 사용자 피드백을 받기 위한 도구다
정답: B
해설: 프로토타입은 완벽할 필요 없어요! "빠르게 만들고 빠르게 배우자"가 원칙!

Q4-5. 음식 배달 서비스 개선 아이디어를 프로토타입하려면?
[A] 실제 배달 시스템을 구축한다
[B] 앱 화면을 종이에 그리고 고객에게 시뮬레이션해본다
[C] 사업계획서를 작성한다
[D] 배달 기사를 먼저 채용한다
정답: B
해설: 실제로 만들기 전에 종이나 간단한 목업으로 먼저 검증! 시간과 비용을 아낄 수 있어요.

STAGE 5: 테스트
═══════════════════════════════════

Q5-1. 프로토타입을 사용자에게 테스트했더니 부정적 피드백이 많다. 어떻게 해야 할까?
[A] 사용자가 이해를 못한 것이므로 설명을 더 한다
[B] 프로토타입을 폐기하고 새로 시작한다
[C] 피드백을 분석해서 문제정의 또는 아이디어 단계로 돌아간다
[D] 다른 사용자 그룹에게 다시 테스트한다
정답: C
해설: 부정적 피드백은 실패가 아니라 배움! 필요하면 앞 단계로 돌아가는 게 디자인씽킹의 핵심이에요.

Q5-2. 테스트 단계에서 가장 중요한 태도는?
[A] 프로토타입이 성공했음을 증명하는 것
[B] 사용자의 반응을 방어 없이 있는 그대로 듣는 것
[C] 사용자가 만족할 때까지 반복 수정하는 것
[D] 최대한 많은 사람에게 보여주는 것
정답: B
해설: 테스트는 "내가 맞다"를 증명하는 게 아니라, 사용자에게서 솔직한 반응을 듣는 시간이에요.

Q5-3. 테스트 결과 "기능은 좋은데 쓸 상황이 없다"는 피드백을 받았다. 어느 단계를 다시 봐야 할까?
[A] 프로토타입 - 더 예쁘게 만든다
[B] 아이데이션 - 다른 아이디어를 낸다
[C] 문제정의 - 사용자가 진짜 필요한 상황을 다시 정의한다
[D] 테스트 - 다른 사용자에게 다시 테스트한다
정답: C
해설: "쓸 상황이 없다"는 문제 자체가 잘못 정의된 거예요. 문제정의로 돌아가야 해요!

Q5-4. 디자인씽킹에서 테스트 후 결과가 좋지 않으면?
[A] 프로젝트를 중단한다
[B] 반복(Iteration)하며 개선한다
[C] 다른 팀에게 넘긴다
[D] 결과 보고서를 작성한다
정답: B
해설: 디자인씽킹은 직선이 아니라 반복! 테스트 → 개선 → 다시 테스트가 핵심이에요.

Q5-5. 테스트할 때 피해야 할 행동은?
[A] 사용자가 프로토타입을 쓰는 모습을 관찰한다
[B] "이 기능은 이렇게 쓰는 거예요"라고 먼저 설명한다
[C] 사용자의 표정과 행동을 기록한다
[D] 열린 질문으로 피드백을 요청한다
정답: B
해설: 사용자가 스스로 탐색하게 두어야 진짜 문제를 발견할 수 있어요. 설명부터 하면 편향된 피드백이 나와요!
